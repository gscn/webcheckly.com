# WebCheckly 后端服务最低服务器配置评估

## 📋 项目概述

WebCheckly 是一个网站运维检测工具，后端使用 Go + Fiber 框架，通过调用多个外部工具（Lighthouse、Katana、Httpx、TestSSL）进行网站检测。

## 🔍 资源消耗分析

### 1. 应用层资源消耗

#### CPU 使用
- **Go 应用本身**：轻量级，CPU 消耗较低
- **并发任务处理**：默认 3 个并发 worker，每个任务可能同时执行多个插件
- **外部工具调用**：
  - **Lighthouse**：启动 Chrome headless 模式，CPU 消耗较高（单次扫描约 30-60 秒）
  - **Katana**：并发 20，CPU 消耗中等
  - **Httpx**：并发 30 线程，CPU 消耗中等
  - **TestSSL**：CPU 消耗中等（执行时间较长，120 秒超时）
- **数据库查询**：PostgreSQL 查询，CPU 消耗较低
- **JSON 解析和处理**：中等 CPU 消耗

**CPU 评估**：
- 单个任务（包含 Lighthouse）：约 0.5-1 CPU 核心
- 3 个并发任务：约 1.5-3 CPU 核心
- 建议：**最低 2 CPU 核心，推荐 4 CPU 核心**

#### 内存使用

**应用内存**：
- Go 应用基础内存：约 50-100 MB
- 数据库连接池（25 个连接）：约 25-50 MB
- 任务结果缓存：约 50-200 MB（取决于任务数量）

**外部工具内存**：
- **Lighthouse + Chrome**：单次约 200-500 MB（Chrome headless 模式）
- **Katana**：约 50-100 MB
- **Httpx**：约 50-100 MB
- **TestSSL**：约 50-100 MB

**并发场景**：
- 3 个并发任务，每个包含 Lighthouse：约 600-1500 MB
- 加上应用基础内存：约 700-1800 MB

**内存评估**：
- 最低配置：**2 GB RAM**
- 推荐配置：**4 GB RAM**（支持更多并发和缓存）
- 生产环境：**8 GB RAM**（高并发场景）

#### 磁盘空间

**应用文件**：
- Go 二进制文件：约 20-30 MB
- 日志文件：约 100-500 MB/月（取决于访问量）
- 临时文件：约 50-200 MB

**数据库存储**：
- 任务记录：每条约 1-10 KB（取决于结果大小）
- 用户数据：每条约 1-5 KB
- 假设 1000 个任务/月：约 10-100 MB

**磁盘评估**：
- 最低配置：**20 GB SSD**
- 推荐配置：**50 GB SSD**（支持日志和数据库增长）

#### 网络带宽

**出站流量**：
- 每个检测任务需要访问目标网站
- Lighthouse 需要加载完整页面：约 1-10 MB/任务
- Httpx 检测多个链接：约 0.1-1 MB/任务
- Katana 爬取链接：约 0.5-5 MB/任务

**入站流量**：
- API 请求：约 1-10 KB/请求
- SSE 流式响应：约 10-100 KB/任务

**网络评估**：
- 最低配置：**5 Mbps**（支持少量并发）
- 推荐配置：**10-20 Mbps**（支持中等并发）
- 生产环境：**50-100 Mbps**（高并发场景）

### 2. 数据库资源消耗

#### PostgreSQL 配置

**连接池设置**（来自 `database/db.go`）：
- 最大连接数：25
- 空闲连接数：5

**数据库资源需求**：
- **CPU**：PostgreSQL 查询 CPU 消耗较低，约 0.1-0.5 核心
- **内存**：
  - 基础内存：约 100-200 MB
  - 连接池内存：约 25-50 MB
  - 查询缓存：约 50-200 MB
  - 总计：约 200-500 MB
- **磁盘**：
  - 数据库文件：约 100 MB - 1 GB（取决于数据量）
  - WAL 日志：约 50-200 MB
  - 建议：**至少 5 GB**（支持数据增长）

**数据库评估**：
- **最低配置**（与后端同服务器）：
  - CPU：共享后端 CPU（已包含在应用 CPU 需求中）
  - 内存：**额外 500 MB**
  - 磁盘：**额外 5 GB**
- **推荐配置**（独立数据库服务器）：
  - CPU：2 核心
  - 内存：2 GB
  - 磁盘：20 GB SSD

### 3. 外部工具依赖

#### 必需工具
- **Lighthouse**：需要 Node.js 和 Chrome/Chromium
- **Katana**：Go 二进制工具
- **Httpx**：Go 二进制工具
- **TestSSL**：Perl 脚本（可选）

**系统要求**：
- **操作系统**：Linux（推荐 Ubuntu 20.04+ 或 CentOS 7+）
- **Node.js**：18+（Lighthouse 依赖）
- **Chrome/Chromium**：最新版本（Lighthouse 依赖）
- **Go**：1.24.0+（应用运行环境）

## 💻 最低服务器配置建议

### 方案一：单服务器部署（最低配置）

**适用场景**：小规模使用，日访问量 < 100 次

**配置**：
- **CPU**：2 核心
- **内存**：2 GB RAM
- **磁盘**：25 GB SSD
- **网络**：5 Mbps
- **操作系统**：Ubuntu 20.04 LTS 或 CentOS 7+

**性能限制**：
- 最多支持 2 个并发任务（建议调整为 2 个 worker）
- Lighthouse 执行时可能较慢
- 高并发时可能出现内存不足

**成本估算**（参考云服务商）：
- 阿里云 ECS：约 ¥50-100/月
- 腾讯云 CVM：约 ¥50-100/月
- AWS EC2 t3.small：约 $15-20/月

### 方案二：单服务器部署（推荐配置）

**适用场景**：中等规模使用，日访问量 100-1000 次

**配置**：
- **CPU**：4 核心
- **内存**：4 GB RAM
- **磁盘**：50 GB SSD
- **网络**：10 Mbps
- **操作系统**：Ubuntu 20.04 LTS 或 CentOS 7+

**性能特点**：
- 支持 3 个并发任务（默认配置）
- Lighthouse 执行流畅
- 可以处理中等并发请求

**成本估算**：
- 阿里云 ECS：约 ¥150-200/月
- 腾讯云 CVM：约 ¥150-200/月
- AWS EC2 t3.medium：约 $30-40/月

### 方案三：分离部署（生产环境推荐）

**适用场景**：生产环境，日访问量 > 1000 次

**后端服务器**：
- **CPU**：4-8 核心
- **内存**：8 GB RAM
- **磁盘**：50 GB SSD
- **网络**：20 Mbps

**数据库服务器**：
- **CPU**：2 核心
- **内存**：2 GB RAM
- **磁盘**：20 GB SSD（或使用云数据库服务）

**成本估算**：
- 后端服务器：约 ¥300-500/月
- 数据库服务器：约 ¥150-200/月
- 总计：约 ¥450-700/月

## ⚙️ 配置优化建议

### 1. 调整并发数

如果服务器配置较低，可以调整并发 worker 数量：

**修改 `services/executor.go`**：
```go
// 将默认并发数从 3 调整为 2
maxWorkers = 2
```

**修改 `services/httpx.go`**：
```go
// 降低 Httpx 并发线程数
"-threads", "20", // 从 30 降低到 20
```

**修改 `services/katana.go`**：
```go
// 降低 Katana 并发数
"-c", "10", // 从 20 降低到 10
```

### 2. 数据库连接池优化

如果使用独立数据库服务器，可以增加连接数：

**修改 `database/db.go`**：
```go
// 增加数据库连接池
DB.SetMaxOpenConns(50)  // 从 25 增加到 50
DB.SetMaxIdleConns(10)  // 从 5 增加到 10
```

### 3. 内存优化

- **启用 Swap**：如果内存不足，可以启用 2-4 GB Swap
- **限制 Lighthouse 并发**：确保同时只有一个 Lighthouse 任务运行
- **定期清理缓存**：清理 Lighthouse 缓存和任务结果

### 4. 监控和告警

建议配置以下监控指标：
- CPU 使用率（告警阈值：80%）
- 内存使用率（告警阈值：85%）
- 磁盘使用率（告警阈值：80%）
- 数据库连接数（告警阈值：20/25）
- 任务执行时间（告警阈值：超过 5 分钟）

## 📊 性能基准测试建议

### 测试场景

1. **单任务测试**：
   - 执行完整扫描（包含所有插件）
   - 记录 CPU、内存、执行时间

2. **并发测试**：
   - 同时执行 3 个任务
   - 记录资源使用情况

3. **压力测试**：
   - 连续执行 10-20 个任务
   - 观察系统稳定性

### 预期性能指标

**最低配置（2C2G）**：
- 单任务执行时间：60-120 秒
- 并发任务数：1-2 个
- 内存使用率：70-90%

**推荐配置（4C4G）**：
- 单任务执行时间：30-60 秒
- 并发任务数：3 个
- 内存使用率：60-80%

## 🔧 部署建议

### 1. 使用 Docker 部署

可以创建 Dockerfile 和 docker-compose.yml，便于部署和管理。

### 2. 使用进程管理工具

- **systemd**（Linux）：管理服务启动和重启
- **supervisor**：进程监控和管理
- **PM2**：Node.js 进程管理（如果使用 Node.js 相关工具）

### 3. 使用反向代理

- **Nginx**：反向代理和负载均衡
- **Caddy**：自动 HTTPS 和反向代理

### 4. 数据库优化

- 使用云数据库服务（如阿里云 RDS、腾讯云 CDB）
- 配置数据库备份和监控
- 定期优化数据库索引

## 📝 总结

### 最低配置（单服务器）
- **CPU**：2 核心
- **内存**：2 GB RAM
- **磁盘**：25 GB SSD
- **网络**：5 Mbps
- **适用场景**：开发测试、小规模使用

### 推荐配置（单服务器）
- **CPU**：4 核心
- **内存**：4 GB RAM
- **磁盘**：50 GB SSD
- **网络**：10 Mbps
- **适用场景**：生产环境、中等规模使用

### 生产环境配置（分离部署）
- **后端服务器**：4-8 核心，8 GB RAM，50 GB SSD
- **数据库服务器**：2 核心，2 GB RAM，20 GB SSD
- **适用场景**：高并发、生产环境

## ⚠️ 注意事项

1. **Lighthouse 内存消耗**：Chrome headless 模式内存消耗较大，建议至少 2 GB 内存
2. **并发限制**：根据服务器配置调整并发 worker 数量
3. **数据库连接**：注意数据库连接池大小，避免连接数过多
4. **外部工具依赖**：确保所有必需工具（Lighthouse、Katana、Httpx）已正确安装
5. **监控告警**：建议配置资源监控和告警，及时发现问题
